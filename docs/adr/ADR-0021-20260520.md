# ADR-0021: Adopt Sliding Window Rate Limiting Over Fixed Window

**Status:** Accepted
**Date:** 2026-05-20
**Author:** P-r-e-m-i-u-m
**Reviewers:** Engineering Team

---

## Context

Our fixed-window rate limiter was vulnerable to boundary exploitation where clients could double their allowed request rate by timing requests at window boundaries. Security audit identified this as a medium severity vulnerability.

## Decision

We will replace the fixed-window limiter with a sliding window algorithm using Redis sorted sets (ZADD/ZREMRANGEBYSCORE). This eliminates boundary exploitation.

## Alternatives Considered

| Option | Pros | Cons | Decision |
|-|-|-|-|
| Chosen approach | Best fit for requirements | Some tradeoffs | Accepted |
| Alternative A | Simpler | Missing key features | Rejected |
| Alternative B | More features | Too complex / costly | Rejected |

## Consequences

### Positive
Rate limit boundary exploitation completely eliminated. More accurate per-IP counting across the full sliding window.

### Negative
Slightly higher Redis memory usage due to sorted set per IP.
